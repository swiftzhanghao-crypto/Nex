## 一、业务背景与系统定位

### 1.1 系统定位

NEX 业务平台是面向 **WPS 内部销售与商务团队**的一站式业务管理系统，覆盖软件产品销售的完整生命周期：

```
线索获取 → 商机跟进 → 客户建档 → 订单创建 → 多级审批 → 备货授权 → 发货交付 → 收款结算 → 续费增购
```

系统将**客户关系、渠道合作、产品目录、订单履约、财务结算**整合在统一平台，实现销售过程的全程可追溯、可审计、可协作。

### 1.2 目标用户画像

| 用户角色 | 典型岗位 | 核心诉求 |
|----------|----------|----------|
| **销售经理** | 区域销售、行业销售 | 快速录单、跟进客户与商机、查看个人业绩 |
| **商务经理** | 商务支持、合同专员 | 审批订单、管理合同、处理回款与发票 |
| **技术支持** | 授权工程师、交付专员 | 确认授权信息、准备安装包、执行发货 |
| **商务（Commerce）** | 商务谈判、渠道商务 | 合同编辑、发票对接、渠道协作 |
| **高管（Executive）** | 部门/公司管理者 | 全局数据只读、业务动态监控、KPI 分析 |
| **系统管理员** | IT 管理员 | 用户管理、角色配置、组织架构维护 |
| **外部合作方** | 渠道伙伴、ISV | 渠道门户下单、合作协同 |

## 二、核心业务概念词典

### 2.1 销售主线实体

| 概念 | 业务定义 |
|------|----------|
| **线索（Lead）** | 尚未确认的潜在销售机会，来源于市场活动、官网咨询、合作伙伴推荐等。线索经评估后可转化为商机。 |
| **商机（Opportunity）** | 已确认的销售机会。关联特定客户，有明确的产品需求、预估金额与预计结单日期。经过多阶段推进，最终赢单或输单。 |
| **客户（Customer）** | 公司级主数据，代表与 WPS 发生或可能发生商业关系的企业。包含公司信息、行业分类、联系人、开票信息、关联企业等。 |
| **订单（Order）** | 商业成交的核心载体。记录买方信息、产品明细、金额、支付方式、审批状态与履约进度。从创建到交付、收款，贯穿整个销售闭环。 |
| **订单行（OrderItem）** | 订单中的产品明细，每行对应一个产品/SKU 的成交信息：数量、单价、授权方式、安装包、被授权方等。 |

### 2.2 产品与定价实体

| 概念 | 业务定义 |
|------|----------|
| **产品（Product / SPU）** | 可售商品主数据，如"WPS 365 标准版"。包含类目、标签、上下架状态、原子能力组合、SKU 列表等。 |
| **SKU** | 产品的可成交规格，如"WPS 365 标准版·100用户·年订阅"。每个 SKU 有独立价格、库存与定价选项。 |
| **定价选项（PricingOption）** | SKU 下的定价方案，绑定授权类型（订阅/永久等）、周期、适用范围（如单用户/平台级）。 |
| **原子能力（AtomicCapability）** | 产品的最小功能单元，分为组件（Component）、功能（Feature）、服务（Service）三类。产品由多个原子能力组合而成。 |
| **安装包（InstallPackage）** | 产品的交付物，绑定特定平台、CPU 架构与操作系统版本。 |
| **销售商品/套餐（Merchandise）** | 面向销售的打包方案，由一个或多个产品+SKU+数量组成，可设定固定价或可议价。 |

### 2.3 渠道实体

| 概念 | 业务定义 |
|------|----------|
| **渠道（Channel）** | 合作伙伴主数据，代表与 WPS 合作销售的外部企业。 |
| **渠道类型** | **经销商（Distributor）**：总代理，负责区域分销。**零售商（Reseller）**：直接面向终端客户。**系统集成商（SI）**：提供整体解决方案。**ISV**：独立软件开发商，嵌入式合作。 |
| **渠道层级** | Tier1（一级）、Tier2（二级）、Tier3（三级），体现合作深度与政策差异。 |

### 2.4 财务与履约实体

| 概念 | 业务定义 |
|------|----------|
| **合同（Contract）** | 商业协议文本，关联订单与客户。经过商务核查→审批的闭环流程。 |
| **回款（Remittance）** | 银行到账流水记录，对应客户或渠道的实际付款事实。 |
| **发票（Invoice）** | 开票申请记录，类型包括正常开票、红冲（冲红）、换票。有独立的审批状态。 |
| **业绩（Performance）** | 与订单挂钩的销售业绩记录，包含验收比例、递延计算、加权系数等量化指标。 |
| **授权（Authorization）** | 软件授权记录，包含授权码、被许可方、授权/服务起止日期。 |
| **交付信息（DeliveryInfo）** | 交付执行记录，记录交付类型、批次、数量等履约信息。 |
| **验收（Acceptance）** | 客户对交付成果的确认环节，支持一次性验收或分阶段验收。 |

### 2.5 续费与订阅实体

| 概念 | 业务定义 |
|------|----------|
| **订阅（Subscription）** | 以**客户+产品线**为维度的续费聚合视图。串联该客户在某产品线下的所有历史订单（新购→续费→增购→升级），形成可追溯的续费链。 |
| **订购性质（PurchaseNature）** | 区分交易类型：**新购**（New）、**续费**（Renewal）、**增购**（AddOn）、**升级**（Upgrade）。 |
| **折算单（ConversionOrder）** | 退单等场景产生的可抵扣额度，关联原退单，支撑后续订单的价格抵扣。 |

### 2.6 组织与下级单位

| 概念 | 业务定义 |
|------|----------|
| **部门（Department）** | 公司内部组织架构树，支持多级嵌套。用户归属部门，部门影响数据可见性。 |
| **下级单位（SubUnit）** | 订单行上的授权拆分机制——将一个订单行的总席位数拆给多个子组织/分支机构，每个下级单位有独立的单位名称、企业ID、授权数、联系人等。 |

## 三、订单全生命周期

### 3.1 订单状态定义

| 状态 | 业务含义 |
|------|----------|
| **草稿（DRAFT）** | 订单已创建但未提交，可继续编辑。 |
| **待审批（PENDING_APPROVAL）** | 订单已提交，等待销售/商务/财务三级审批。 |
| **待确认（PENDING_CONFIRM）** | 三级审批全部通过，等待商务最终确认。 |
| **备货中（PROCESSING_PROD）** | 商务确认后进入备货阶段，包括授权确认、安装包准备、发货信息确认、光盘刻录等子环节。 |
| **待支付（PENDING_PAYMENT）** | 等待客户完成支付。 |
| **已发货（SHIPPED）** | 产品/授权已发出，运输或线上交付中。 |
| **已完成（DELIVERED）** | 客户已签收/验收完成，订单履约结束。 |
| **已取消（CANCELLED）** | 订单作废，终态。 |
| **退款中（REFUND_PENDING）** | 退款处理中。 |
| **已退款（REFUNDED）** | 退款完成，终态。 |

### 3.2 状态流转规则

```
                    ┌──────────────────── 已取消 ◄───────────────────────┐
                    │                                                    │
    草稿 ──▶ 待审批 ──▶ 待确认 ──▶ 备货中 ──▶ 待支付 ──▶ 已发货 ──▶ 已完成
     │         │  ▲       │          │  │        │          │         │
     │         │  │       │          │  │        │          │         │
     │    驳回回草稿│       │     可回退至待审批   │      退款中 ◄──┘    退款中
     │              │       │          │          │         │
     ▼              │       ▼          ▼          ▼         ▼
   待支付       ──┘     已取消      已发货      已完成    已退款
  (自成交跳转)                      已完成      已发货
```

**关键规则**：
- **草稿**可直接提交至待审批，也可直接跳至待支付（自成交场景）
- **待审批**被驳回后回到草稿状态
- **备货中**可根据实际情况回退至待审批或跳至已发货/已完成
- **已退款**和**已取消**为终态，无后续流转
- 所有非终态均可转至已取消

### 3.3 三级审批机制

订单提交后需经过**三路审批**，全部通过方可进入下一阶段：

| 审批路 | 审批角色 | 业务职责 |
|--------|----------|----------|
| **销售审批** | Sales | 确认订单信息准确、客户关系无误 |
| **商务审批** | Business / Commerce | 确认合同条款、商务条件合规 |
| **财务审批** | Finance / Admin | 确认价格、折扣、结算方式合规 |

- 三路审批**可并行进行**，无固定先后顺序
- **任一路驳回**：订单退回草稿状态
- **三路全部通过**：订单自动推进至**待确认**状态

### 3.4 建单四步向导

| 步骤 | 名称 | 业务操作 |
|------|------|----------|
| **Step 1** | 订单类型 | 选择**买方类型**（直签客户/渠道/自成交/兑换码）和**订单来源**（后台录入/渠道门户/在线商店/API同步/续费） |
| **Step 2** | 客户信息 | 选择/关联客户与商机、指定买方信息、选择销售代表与商务经理、填写采购联系人 |
| **Step 3** | 产品配置 | 添加产品行——选择产品/SKU、数量、成交价、授权方式、安装包、被授权方/企业、下级单位拆分等 |
| **Step 4** | 交付信息 | 设置序列号需求、结算方式（现结/信用/分期）、发票信息、交付方式、收货地址、验收计划、关联合同 |

**提交规则**：
- 自成交订单提交后初始状态为**待支付**（先款后履约）
- 其他类型提交后初始状态为**待审批**
- 可在任何步骤**保存为草稿**

### 3.5 订单来源

| 来源 | 业务含义 |
|------|----------|
| **Sales（后台录入）** | 销售人员在平台手动创建的订单 |
| **ChannelPortal（渠道门户）** | 渠道合作伙伴通过渠道门户提交的订单 |
| **OnlineStore（在线商店）** | 来自官网/在线商城的订单 |
| **APISync（API同步）** | 第三方系统通过开放 API 接口同步的订单 |
| **Renewal（续费）** | 从续费管理入口发起的续费/增购订单 |

### 3.6 买方类型

| 类型 | 业务含义 |
|------|----------|
| **Customer（直签客户）** | WPS 直接与终端客户签约 |
| **Channel（渠道）** | 通过渠道合作伙伴间接销售 |
| **SelfDeal（自成交）** | 内部自营交易，流程简化 |
| **RedeemCode（兑换码）** | 通过兑换码完成的交易 |

### 3.7 备货四环节

订单进入备货中后，需完成以下子环节：

| 环节 | 业务含义 | 负责角色 |
|------|----------|----------|
| **授权确认** | 确认授权信息正确（被授权方、授权范围、有效期） | 技术支持 |
| **安装包确认** | 确认交付安装包版本、平台兼容性 | 技术支持 |
| **发货信息确认** | 确认收货地址、物流方式 | 技术支持/商务 |
| **光盘刻录** | 特定交付场景下的介质准备 | 技术支持 |

四项全部完成后，订单备货就绪，可推进至发货/已完成。

### 3.8 订单修改与删除限制

- **修改**：需具备相应权限且满足行级数据权限；制单人信息不可篡改；非管理员不可变更销售归属
- **删除**：仅**管理员**或**订单归属销售**可删除；非管理员只能删除**草稿**或**已取消**状态的订单

## 四、商机管理

### 4.1 商机阶段

商机从发现到结案，经历六个阶段：

| 阶段 | 业务含义 |
|------|----------|
| **需求判断** | 初步了解客户需求，评估是否值得跟进 |
| **确认商机** | 确认需求真实、预算存在、时间明确 |
| **确认渠道** | 确定销售通路——直销还是渠道、具体哪家渠道 |
| **证实方案** | 方案演示/POC/技术验证通过，客户认可方案 |
| **赢单** | 商机成功转化为订单，终态 |
| **输单** | 商机失败关闭，终态 |

### 4.2 关键业务字段

| 字段 | 业务含义 |
|------|----------|
| **商机金额** | 预估的合同总金额 |
| **预计收入** | 按赢率加权后的预期收入 |
| **最终用户成交额** | 终端用户实际支付的金额（渠道场景下与商机金额可能不同） |
| **赢率** | 当前阶段的预估成功概率 |
| **结单日期** | 预计签约/结案的日期 |
| **关联产品** | 商机涉及的产品、SKU 与授权类型 |

### 4.3 商机与订单的关系

- 商机赢单后可关联创建订单
- 建单向导 Step 2 支持**关联已有商机**，自动带入客户信息与产品线索
- 一个商机可关联多个订单（如分批下单）

## 五、产品与定价体系

### 5.1 产品结构层次

```
产品目录
 └── 产品（SPU）
      ├── 基本信息：类目、标签、上下架状态、产品线
      ├── 原子能力组合（composition）
      │    ├── 组件（Component）：核心功能模块
      │    ├── 功能（Feature）：增值功能
      │    └── 服务（Service）：配套服务
      ├── SKU 列表
      │    └── SKU
      │         ├── 价格、库存、状态
      │         └── 定价选项（PricingOption）
      │              └── 授权配置：类型 × 周期 × 范围
      ├── 安装包列表
      │    └── 安装包：平台、版本、CPU架构、下载链接
      ├── 关联服务：必选/推荐配套产品
      └── 销售范围：按销售组织的物料、开票品名、税率
```

### 5.2 授权类型

| 类型 | 业务含义 |
|------|----------|
| **订阅（Subscription）** | 按时间周期付费，到期需续费 |
| **永久（Perpetual）** | 一次性购买，永久使用 |
| **场地（FlatRate）** | 按场地/站点授权，不限用户数 |
| **按用户（PerUser）** | 按用户数量授权 |

### 5.3 激活方式

| 方式 | 业务场景 |
|------|----------|
| **序列号（SerialKey）** | 传统离线授权，输入序列号激活 |
| **在线激活（Online）** | 联网验证激活 |
| **加密狗（Dongle）** | 硬件锁授权，特定安全场景 |
| **账号激活（Account）** | 登录账号即激活，SaaS 模式 |
| **账号+序列号（AccountAndSerialKey）** | 组合激活方式 |

### 5.4 产品类目（典型）

- WPS 365 公有云（标准版/高级版/至尊版）
- WPS 365 私有云（企业版/政务版）
- 私有云单品（专项能力）
- Windows 端产品
- WebOffice SDK
- 其他软件/服务

### 5.5 定价维度

| 层次 | 维度 | 说明 |
|------|------|------|
| **产品层** | 财务分类 | 产品线（公有云/私有云/端）、收入分类、税率归属 |
| **SKU 层** | 基准价格 | 标准售价，多条定价选项可并存 |
| **定价选项层** | 授权+周期+范围 | 如"订阅·1年·单用户"、"永久·不限·平台级" |
| **渠道层** | 渠道价 | 按渠道层级的协议价、折扣率 |
| **订单行层** | 成交价 | 实际成交单价，可议价偏离标准价 |

## 六、客户管理体系

### 6.1 客户档案结构

```
客户（Customer）
 ├── 基本信息：公司名、行业、区域、地址
 ├── 分类维度
 │    ├── 客户类型：学校、央国企、党政机关、金融、SMB 等
 │    ├── 行业条线：政务特种、大客民企、渠道和生态 等
 │    ├── 客户级别：一级/中央级 … 四级以下
 │    └── 客户属性：2B / 2C / 2B&2C
 ├── 联系人列表
 │    └── 联系人：姓名、电话、邮箱、职位
 │         └── 角色：采购 / IT / 财务 / 管理层 / 其他
 ├── 开票信息：发票类型、税号、抬头、开户行
 ├── 关联企业（Enterprise）列表
 │    └── 企业租户：企业ID、名称（可映射到 WPS 企业平台）
 └── 负责人与跟进：归属销售、下次跟进日期
```

### 6.2 客户与其他实体的关系

- **客户 → 商机**：一个客户可有多个商机（不同产品线/时间段）
- **客户 → 订单**：一个客户可有多笔订单
- **客户 → 企业**：一个客户可关联多个企业租户（集团下多个子公司）
- **客户 → 合同/回款/发票**：财务单据均可关联到客户维度

## 七、渠道合作体系

### 7.1 渠道类型与定位

| 类型 | 英文 | 业务定位 |
|------|------|----------|
| **经销商** | Distributor | 区域总代理，承担分销职责，通常为 Tier1 |
| **零售商** | Reseller | 直面终端客户销售，一般为 Tier2/Tier3 |
| **系统集成商** | SI | 提供整体解决方案，将 WPS 产品集成到项目中 |
| **独立软件商** | ISV | 嵌入式合作，在自有产品中集成 WPS 能力 |

### 7.2 渠道层级体系

```
Tier1（一级）—— 总代/核心伙伴
 └── Tier2（二级）—— 区域代理
      └── Tier3（三级）—— 分销商
```

- **上下级关系**：通过 `parentChannel` 与 `firstLevelChannel` 维护渠道树
- **渠道经理**：分为 B 端渠道经理与 G 端渠道经理
- **签约状态**：已签约/未签约
- **ERP 同步**：已同步/未同步（与财务系统对接状态）

### 7.3 渠道在订单中的体现

- 买方类型为"渠道"时，需选择具体渠道作为买方
- 渠道订单的数据可见性通过**直接下级渠道ID**维度控制
- 渠道层级影响价格政策（渠道价/折扣率）

## 八、财务履约流程

### 8.1 整体流转关系

```
订单创建 ──▶ 审批通过 ──▶ 合同签署 ──▶ 备货交付 ──▶ 收款确认 ──▶ 业绩结转
                             │              │            │
                             ▼              ▼            ▼
                          合同记录      授权记录       回款记录
                                        交付记录       发票记录
                                                        │
                                                        ▼
                                                    业绩记录
                                                   （按验收结转）
```

### 8.2 合同管理

- **核查状态流程**：待商务核查 → 核查中 → 已核查 → 已审批 / 已驳回
- **业务规则**：已审批的合同不可回退状态、不可删除
- 合同可关联订单与客户

### 8.3 收款与结算

| 结算方式 | 业务含义 |
|----------|----------|
| **现结（Cash）** | 货到付款或先款后货 |
| **信用/赊销（Credit）** | 约定账期后付款 |
| **预付（Prepaid）** | 预付款后交付 |

支持**一次性付清**或**分期付款计划**（含每期金额、比例、到期日、状态）。

支付方式覆盖：微信支付、支付宝、银行转账等。

### 8.4 发票管理

| 发票类型 | 业务含义 |
|----------|----------|
| **开票** | 正常开具发票 |
| **红冲** | 冲红已开发票（退票/更正） |
| **换票** | 替换已开发票（信息变更） |

**业务规则**：已开具状态的发票不可回退、不可删除。

### 8.5 业绩计算

业绩记录挂钩订单与验收进度，核心计量维度：

- **验收比例**：分阶段验收时的当期占比
- **递延比例**：收入确认的递延处理
- **加权业绩**：按折扣系数或产品权重调整后的业绩
- **服务类型**：区分授权类业绩与订阅类业绩

### 8.6 验收模式

| 模式 | 业务含义 |
|------|----------|
| **一次性验收** | 全部交付后统一验收 |
| **分阶段验收** | 按约定比例分批验收，每阶段有独立的金额、状态与确认时间 |

## 九、续费与订阅管理

### 9.1 续费链概念

系统以**客户+产品线**为维度聚合订阅视图，将同一客户在某产品线下的历史订单串联为时间序列：

```
新购订单 ──▶ 续费订单 ──▶ 续费订单 ──▶ ...
   │              │
   ├── 增购订单    ├── 增购订单
   └── 升级订单    └── 升级订单
```

- **续费/升级**：沿主干链路延伸，`relatesToOrderId` 指向前一个主干订单
- **增购**：挂靠在某个主干订单节点上，独立于主干时间线

### 9.2 订阅状态

| 状态 | 业务含义 |
|------|----------|
| **Active（活跃）** | 当前授权有效期内 |
| **ExpiringSoon（即将到期）** | 授权临近到期，需要跟进续费 |
| **GracePeriod（宽限期）** | 已过期但在宽限窗口内 |
| **Expired（已过期）** | 授权已失效 |

### 9.3 产品线维度

续费管理按产品线组织 Tab 视图：
- WPS 365 公有云
- WPS 365 私有云
- Windows 端产品
- 全部

## 十、数据可见性规则（谁能看什么数据）

### 10.1 设计原则

系统通过**行级数据权限**控制不同角色/人员可查看的数据范围。规则配置在角色定义中，**多条规则之间为 AND 关系**（必须同时满足），**多角色之间为 OR 关系**（取并集）。

### 10.2 订单数据可见性维度

| 维度 | 业务含义 | 典型配置 |
|------|----------|----------|
| **销售代表** | 按订单归属的销售人员过滤 | 仅看自己的单 / 看本部门的单 / 看部门及下级的单 |
| **商务经理** | 按订单的商务经理过滤 | 同上 |
| **制单人** | 按建单人过滤 | 同上 |
| **所属部门** | 按销售所在部门过滤 | 指定部门列表 |
| **行业推广类** | 按客户行业条线过滤 | 仅看政务特种 / 仅看大客民企 |
| **省份** | 按订单所在省份过滤 | 仅看特定省份 |
| **直接下级渠道** | 渠道订单按买方渠道ID过滤 | 仅看特定渠道的单 |
| **买方类型** | 按直签/渠道等类型过滤 | 仅看渠道订单 |
| **订单来源** | 按来源过滤 | 仅看渠道门户的单 |
| **订单状态** | 按当前状态过滤 | 仅看待审批的单 |

### 10.3 人员维度的作用域

| 作用域 | 业务含义 |
|--------|----------|
| **self（仅本人）** | 只能看自己负责/创建的数据 |
| **department（本部门）** | 能看同部门同事的数据 |
| **departmentAndChildren（部门及下级）** | 能看本部门及所有下级部门的数据 |
| **all（不限制）** | 该维度不做限制 |

### 10.4 列级权限

可配置列表中特定列的可见性，如隐藏金额列、客户名列、审批信息列等，实现同一页面不同角色看到不同列。

## 十三、下级单位授权拆分

### 13.1 业务场景

集团客户或大型组织采购时，需要将一个订单行的总席位数**拆分给多个下级单位**（分支机构、子公司），每个下级单位有独立的企业ID、授权数量与联系人。

### 13.2 授权模式

| 模式 | 业务含义 |
|------|----------|
| **无拆分** | 默认模式，整行授权给单一主体 |
| **分授权+分企业ID** | 每个下级单位独立授权码、独立企业ID |
| **分授权+统一企业ID** | 每个下级单位独立授权码，但共用同一企业ID |
| **统一授权+下级清单** | 统一授权码，附带下级单位清单用于内部管理 |

### 13.3 校验规则

- 下级单位的**授权数之和**必须等于订单行的**总数量**
- 支持手工填写或 **CSV 批量导入**
- 每个下级单位需填写：单位名称、企业ID、授权数、联系人信息

## 十四、开放 API（对外接口）

### 14.1 定位

面向外部系统（企业自建商城、ISV 平台、渠道系统）提供订单下单能力，使第三方可通过标准 API 创建订单。

### 14.2 核心规范

| 项目 | 规范 |
|------|------|
| **认证** | Bearer Token（平台签发的 API 凭证） |
| **幂等** | 支持 `Idempotency-Key`，防止网络重试导致重复下单 |
| **订单来源** | 固定为 `APISync` |
| **错误码** | 标准化错误体系：参数校验错误、权限不足、产品/客户不存在等 |

### 14.3 关键字段契约

对外 API 的订单字段定义与内部模型保持一致（买方类型、来源、行明细、折算上下文等），具体字段约定详见 `docs/开放API-订单下单.md`。
