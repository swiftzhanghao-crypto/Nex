## 一、产品定位

**NEX 业务平台**（又称 WPS 365 业务平台）是面向 **WPS 内部销售与商务团队** 的一站式管理系统，覆盖从线索获取到订单交付的完整业务链路：

```
线索 → 商机 → 订单创建与审批 → 备货/交付 → 续费/增购 → 客户档案
```

**核心价值**：将销售过程、客户关系、产品主数据、财务履约、渠道管理整合在统一平台，配合基于角色的功能权限与行/列级数据权限，支撑不同岗位人员的日常协作。

## 四、数据模型概览

### 4.1 数据库表（SQLite，15 张核心表）

| 表 | 说明 | 关键字段 |
|----|------|----------|
| `users` | 用户 | `role`（JSON 数组）、`department_id`、`password_hash` |
| `departments` | 部门树 | `parent_id` 自引用 |
| `roles` | 角色定义 | `permissions`、`row_permissions`、`row_logic`、`column_permissions`、`app_permissions`（均 JSON） |
| `customers` | 客户 | `contacts`/`enterprises`/`billing_info`（JSON）；`owner_id` |
| `products` | 产品 | `skus`/`composition`/`install_pkgs`/`rights`（JSON） |
| `channels` | 渠道 | 类型、级别、区域、合同状态 |
| `opportunities` | 商机 | `customer_id` FK → customers |
| `orders` | 订单 | `customer_id` FK → customers；`items`/`approval`/`extra`（JSON） |
| `contracts` | 合同 | 关联订单/客户 |
| `remittances` | 回款 | 回款记录 |
| `invoices` | 发票 | 发票记录 |
| `performances` | 业绩 | 销售业绩 |
| `authorizations` | 授权 | 软件授权记录 |
| `delivery_infos` | 交付信息 | 交付/物流 |
| `audit_logs` | 审计日志 | 操作人/目标/变更快照 |
| `spaces` | 应用 | 独立权限树、资源维度、列配置 |
| `space_roles` | 应用角色 | 应用内角色定义 |
| `space_members` | 应用成员 | `(space_id, user_id)` UNIQUE |

### 4.2 核心实体关系

```
Department ──< User（多对一，department_id）
User ──< Order（制单人 / 销售人员）
Customer ──< Order（customer_id）
Customer ──< Opportunity（customer_id）
Order ──< OrderItem（JSON items 数组，引用 Product/SKU 快照）
Order ──< Contract / Remittance / Invoice / Performance / Authorization / DeliveryInfo
Product ──< ProductSku（JSON skus 数组）
Product ──< AtomicCapability（JSON composition 数组）
Space ──< SpaceRole ──< SpaceMember
User ──>< Role（users.role JSON 数组引用 roles.id）
```

### 4.3 前端核心类型（`types.ts`）

关键领域对象：

- **User / RoleDefinition / Department**：用户、角色（含权限矩阵）、组织
- **Product / ProductSku / SalesMerchandise**：产品、SKU、可售套餐
- **Customer / CustomerContact / BillingInfo / Enterprise**：客户全貌
- **Opportunity / OpportunityProduct**：商机与关联产品
- **Order / OrderItem / OrderApproval / SubUnit / SerialNumber**：订单全生命周期
- **Contract / Remittance / Invoice / Performance / Authorization / DeliveryInfo**：财务履约
- **Subscription / SubscriptionOrderRef**：续费链
- **Space / SpaceRole / SpaceMember**：应用权限体系

## 五、权限体系

### 5.1 功能权限（RBAC）

- **角色**：Admin、Sales、Business、Technical、Executive、Commerce 等
- **权限矩阵**：`resource:action` 映射到允许的角色列表
  - 后端：`server/rbac.ts` / `backend/internal/rbac/rbac.go`
  - 前端：`hooks/usePermission.ts`
- **权限树**：四级结构，约 99 个权限点（详见 `docs/功能权限清单.md`）
- **路由守卫**：`RequireAnyPermission` 组件保护受限页面

### 5.2 行级数据权限

- **存储**：`roles.row_permissions` JSON 数组，每条规则含 `resource`、`dimension`、`values`
- **维度**（以订单为例）：
  - 人员：自己、部门、部门及下级
  - 来源：直销/渠道
  - 状态：各阶段
  - 行业线、省份
  - 渠道 ID（渠道单独控制）
- **执行**：
  - 列表查询：SQL WHERE 下推（`buildRowPermissionWhere`）
  - 单条校验：`checkRowPermissionForSingle`
  - 多规则合并：AND 逻辑
- **多角色**：用户可拥有多角色，行权限取**并集**
- 详见 `docs/订单行权限维度说明.md`

### 5.3 列级权限

- **存储**：`roles.column_permissions` JSON
- **作用**：控制列表/详情中特定字段的可见性

### 5.4 应用权限（Spaces）

- **独立权限树**：每个 Space 有自己的权限定义
- **应用角色**：`SpaceRole` 独立于全局角色
- **成员管理**：`SpaceMember` 关联用户与应用角色
- **存储**：全局角色的 `app_permissions` JSON 中按 Space ID 存片段

## 六、API 架构

### 6.1 认证

- **方式**：JWT Bearer Token
- **密码**：scrypt + salt，兼容旧 SHA-256 并在登录时自动升级
- **开发环境**：所有用户密码 `123456`
- **`JWT_SECRET`**：生产环境必须 ≥16 字符，否则拒绝启动

### 6.2 路由清单

|------|------|----------|
| `/api/health` | 健康检查 | `GET` |

| `/api/users` | 用户 | CRUD + 角色 CRUD + 部门/角色元数据 |

1. 本地 `npm run build` → `dist/`
2. 打包（tar.gz，不含 `node_modules`）
3. SCP 上传至服务器
4. 远程解压 → `npm install --omit=dev` → `npm rebuild better-sqlite3`
5. 重启计划任务 `schtasks /run /tn NexSrv`
6. 验证：`/api/health` + 前端页面

详见 Cursor Skill `deploy-system/SKILL.md`。

## 十一、关键设计决策与已知事项

### 11.1 双后端共存

当前项目同时存在 Node.js（`server/`）和 Go（`backend/`）两套后端实现：
- **Node.js/Express**：当前生产运行栈，功能完整
- **Go/Gin**：重构目标，API 接口与 Node 版保持一致
- 两者共享同一 SQLite 文件与 schema，密码哈希格式兼容

### 11.2 Mock vs API 模式

前端通过 `VITE_API_MODE` 环境变量切换：
- `true`：走真实后端 API（生产、联调）
- `false`：使用 `data/staticData.ts` + `data/generators.ts` 的本地 Mock 数据（独立开发、演示）

### 11.3 生产环境 AI 路由

`server/production.ts` 未注册 `aiRoutes`，AI 助手在生产环境自动探测 `/api/ai/status` 后降级为本地模式。如需在生产启用服务端 AI，需在 `production.ts` 中补充挂载。

### 11.4 文档与实现一致性

部分文档撰写时间早于代码迭代，可能存在差异：
- `README.md` 中的 `frontend/` 目录 vs 实际根目录直接放前端代码
- `产品说明文档.md` 中 React 18 vs 实际 React 19
- `AI业务助手技术文档.md` 中 `VITE_GEMINI_API_KEY` vs 实际 `GEMINI_API_KEY`

以代码实现为准，文档供参考。
