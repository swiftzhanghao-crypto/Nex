# 场景与集成说明

## 场景一：给现有业务接口增加大模型能力

目标：在不改动现有业务主流程的前提下，引入可配置的模型调用。

建议做法：
- 在 Service 层封装 `Query(ctx, prompt, data, modelName)`。
- 由 Controller 只负责参数透传与返回结构，不承担模型细节。
- 通过 `modelName` 支持请求级覆盖；为空时使用配置默认模型。

参考示例：
- `examples/llm_service_example.go`
- `examples/llm_api_controller_example.go`

## 场景二：希望模型尽量返回结构化 JSON

目标：减少自然语言输出导致的解析成本。

建议做法：
- 在 system prompt 中强约束“只返回 JSON 对象”。
- 在服务侧实现两段解析：
  - 直接解析完整文本 JSON。
  - 回退解析 markdown code fence 中的 JSON 片段。
- 解析失败时保留原始 content，避免业务彻底失败。

参考示例：
- `examples/llm_service_example.go` 中 `tryParseJSON`

## 场景三：接入统一 AI Gateway 并提升可运维性

目标：在网关治理下可观测、可降级、可切模。

建议做法：
- 配置集中在 `llm.gateway.*`，避免散落硬编码。
- 为 HTTP Client 统一超时时间与重试参数。
- Header 中按需透传网关维度字段（如 uid/product/intention）。
- 日志中输出状态码与截断后的错误 body，避免打印完整敏感信息。

参考示例：
- `examples/config.llm.example.yaml`
- `examples/llm_service_example.go` 中 `setHeaders`、`truncateText`

## 集成边界建议

- 若只需要“单次问答 + JSON 结果”，优先用本技能最小实现，避免过早引入重型 Agent 框架。
- 若后续出现“多步骤推理/工具调用/工作流编排”，再叠加 Eino 相关技能，不要把编排逻辑硬塞进基础网关层。
