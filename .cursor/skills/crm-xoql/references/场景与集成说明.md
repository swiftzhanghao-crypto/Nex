# 场景与集成说明

## 场景一：从零接入 CRM XOQL

目标：最短路径打通 token 与查询链路。

建议做法：
- 在服务配置中声明 `crm.xiaoshouyi` 配置段，字段以 `examples/config.crm.example.yaml` 为基线。
- 落地一个独立 `XSYService`，先只实现 `GetAccessToken`、`QueryXOQL`、`QueryXOQLPaged` 三个能力。
- 暴露一个 demo 接口（见 `examples/risk_board_demo_controller.go`），接收 SQL 并返回记录数和 records。

验收标准：
- 能成功拿到 token，且缓存命中时不重复请求 token 接口。
- 单页查询 `code` 返回 200（兼容字符串与数字）。
- 超过 3000 条结果时自动分页聚合。

## 场景二：已有 CRM 接入，准备标准化重构

目标：把零散逻辑收敛为稳定基础层，降低业务模块重复实现。

建议做法：
- 将 token 获取逻辑集中到 `GetAccessToken`，并加入 23h 预过期缓存策略。
- 将请求头、URL 拼接、错误处理统一在 `QueryXOQL`，业务层不直接发 HTTP。
- 将分页查询统一为 `QueryXOQLPaged`，避免每个模块都手写 `limit offset,size` 循环。

验收标准：
- 业务层仅传 XOQL 与是否管理视角参数，不关心 HTTP 细节。
- 所有 CRM 错误日志包含 `code`、`msg` 和原始查询语句摘要。

## 场景三：业务代码只保留 demo，不迁移旧看板口径

目标：避免把特定业务模型（风险看板）硬编码进新项目。

建议做法：
- 仅保留一个 demo SQL 执行接口，作为集成样板。
- demo 输入限制为只读查询语句；禁止 DML/DDL。
- 后续业务服务基于 demo 能力逐步扩展，不直接照搬 `risk_board.go` 的聚合逻辑。

参考示例：
- `examples/risk_board_demo_controller.go`

## 与原仓库内容的关系

- 该技能抽取的是通用 CRM 基础能力，而不是业务领域规则。
- 业务字段命名、枚举值、统计口径请在接入项目中重新确认。
- 如需引入白名单、组织权限、复杂统计，请在项目内单独设计服务层，不放入此基础技能。
